## Прежде чем писать код

Прочитай (это источники правды, не выдумывай сверх них):
- `AGENTS.md` — конвенции разработки и карта документации.
- `GOAL.md` — цели и non-negotiable ограничения.
- `docs/ARCHITECTURE.md` — слои, каталоги `src/`, правила зависимостей. Обнови его, если двигаешь
  файлы/каталоги или меняешь зависимости между слоями.
- `docs/TESTING.md` — как тестировать каждый слой. Пиши тесты по этим правилам.
- Профильные доки в `docs/TODO/` и `docs/` по теме issue.

## Правила фичи (обязательно учитывай)

1. **Api extensions.** Часто мало реализовать поведение — надо ещё раскомментировать/дореализовать
   соответствующие методы поверхности расширений в `src/Extensions/**` (`vscode.d.ts` и хосты в
   `src/Extensions/Host/**`). Если фича этого касается — доведи и покрой тестами.
2. **Бинды и цвета — из vscode, через настройки/палитры.** Клавиши и цвета не хардкодим: берём
   дефолты как в VS Code и заводим через систему настроек и цветовые палитры/темы. Никаких
   магических значений по месту.
3. **Неочевидное поведение → как в VS Code.** Если в issue не описан крайний случай — воспроизводи
   поведение VS Code.
4. **Доп. контролы → подфичи.** Если фича требует новых контролов/виджетов и целиком не умещается —
   разбей на подфичи и **сделай хотя бы первую** (осмысленную, самодостаточную), остальное опиши.
5. **Стиль репозитория:** Conventional Commits (`type(scope): subject`; `feat`/`fix`/`perf` — только
   про видимое пользователю поведение редактора; инфраструктура — `refactor`/`test`/`chore`/`ci`/…).
   Не чини линтер. Расширения в путях импортов обязательны. Приватные поля — без подчёркивания.
   Тестовые файлы не раздувай, дели по подтемам (`Foo.Events.test.ts`).

## Оцени объём и выбери ветку действий

- **Умещается в нашу модель** (понятно, архитектура не ломается, крайние случаи покрываемы):
  реализуй полностью, добавь тесты, при необходимости — методы Api extensions. Затем **открой PR**
  (см. ниже).
- **Слишком большое / ломает архитектуру / нужно решение владельца, которое ты не можешь принять
  сам** (упущено что-то важное, требуется переделка архитектуры): **не пиши половину кода**.
  Вместо этого оставь разбор комментарием в issue (`gh issue comment {{ISSUE_NUMBER}} -F <file>`):
  что за задача на самом деле, почему большая, предлагаемое разбиение на подфичи, риски, открытые
  вопросы к владельцу. После этого остановись (PR не открывай).
- **Промежуточный случай** (эпик, но первая подфича умещается): сделай первую подфичу + PR, а в теле
  PR и/или комментарии к issue опиши оставшиеся подфичи.

## Как открыть PR (если реализовал)

1. Убедись, что тесты по `docs/TESTING.md` зелёные (`npm run` / `npx vitest`), `npx tsc` без ошибок.
2. Коммит(ы) по Conventional Commits.
3. `git push -u origin {{BRANCH}}`.
4. `gh pr create --draft --base main --head {{BRANCH}} --title "<conventional title>" --body <...>`.
   В теле PR: ссылка на issue (`Closes #{{ISSUE_NUMBER}}` для полностью закрытых; для частичных —
   `Refs #...` + что осталось), краткое описание изменений и как проверить.

## Обязательный финал (не забудь!)

Что бы ни случилось — PR открыт, разбор оставлен комментарием, или ты не смог доделать — **самым
последним действием выполни ровно эту команду:**

```
touch '{{SENTINEL}}'
```

Это сигнал оркестратору, что слот освободился и можно запускать следующую задачу. Без него
следующий воркер не стартует. Сессия при этом остаётся открытой — владелец может подойти и посмотреть.

Действуй. Доведи задачу до PR или до разбора-комментария — не останавливайся посередине.
