# Diff — дифф-редактор и вкладка Changes

Цель: показывать изменения файлов — сначала diff-редактором, затем списком изменённых
файлов (аналог SCM-вьюхи VS Code, одно репо, без worktree и мультирепо).

## Как соотносятся дифф и Changes

Это не две фичи, а три слоя. «Сравнить два файла» и «дифф из Changes» — **один и тот же
виджет**; отличается только то, откуда берётся левая сторона: при сравнении файлов она с
диска, а в Changes — версия из `HEAD`, которой на диске нет. Вот эта виртуальная левая
сторона и есть настоящая граница между фичами.

Показательно, что в API VS Code нет способа отдать редактору готовый дифф. Есть только
`QuickDiffProvider.provideOriginalResource(uri): Uri` — расширение отдаёт **URI исходной
версии**, то есть контент, а дифф считает workbench своим алгоритмом. Это подтверждается и
реализацией: `contrib/scm/browser/quickDiffModel.ts` резолвит оригинал как текстовую модель
(`createModelReference`) и пересчитывает дифф по каждому `onDidChangeContent`. Разбора
вывода `git diff` там нет нигде.

## Кто считает дифф

Разделяем **change set** (какие строки изменились — алгоритм) и **diff view model**
(выровненные пары строк, filler-строки, вид каждой строки, маппинг view→сторона+doc-строка).
Второе — наше всегда. Первое — порт с двумя поставщиками, по признаку «файл открыт»:

| | Файл открыт в редакторе | Файл не открыт (коммит, превью из Changes) |
|---|---|---|
| Обе стороны | буфер уже в памяти, цена принята при открытии | не материализуем никогда |
| Change set | наш вычислитель, живой при наборе | ханки от `git diff` |
| Большой файл | размер уже принят фактом открытия | ничего не читаем, только ханки |

Почему не парсить `git diff` всегда:

- **Шорткат перестаёт быть шорткатом там, где начинается фича.** Для side-by-side нужны оба
  текста, то есть `git show HEAD:path` зовётся всё равно; а имея два буфера, посчитать дифф
  самим дешевле, чем восстанавливать старую сторону из тел ханков (`\ No newline at end of
  file`, rename detection, бинарники, свои правила EOL/`.gitattributes`).
- **Дифф от диска отстаёт от буфера.** `extensions/git/main.ts` перечитывает ханки по
  `onDidSaveTextDocument` — change-bars в гуттере залипшие, пока идёт набор. В VS Code они
  живые.
- **Git не умеет остальные случаи**: два произвольных файла, буфер против диска,
  untitled-буфер, файл вне репозитория.
- **Нет intra-line.** `--word-diff` — другой формат вывода, склеивать с построчным неудобно.

Что остаётся за git безоговорочно: список изменённых файлов и статусы, контент любой
ревизии, **detection переименований** (`git diff -M` — это парование путей, уровень списка
файлов, своим построчным диффом не получить).

## Форма вью: строки и свёрнутые куски

Вью — упорядоченный список строк, где неизменённый кусок можно схлопнуть в плейсхолдер
«⋯ N строк». Между ханками хранится не контент, а количество скрытых строк.

**Важно (была ошибка в этом разделе):** свёртка НЕ является дефолтом VS Code. И в
запинненном `1.127.0`, и в текущем main `editor/common/config/diffEditor.ts` даёт
`hideUnchangedRegions.enabled: false` (`contextLineCount: 3`, `minimumLineCount: 3`,
`revealLineCount: 20`) — по умолчанию показывается весь файл. Раньше здесь было записано
обратное; на дизайн это не влияет, но на выбор дефолта — влияет, и мы берём
upstream'овский.

Зачем тогда свёртка: чтобы **не материализовать файл целиком** там, где это дорого или
негде — превью из списка изменений, просмотр коммита, — плюс в терминале строк на экране
мало. Реализовано так, что ленивость получается даром: модель оперирует только номерами
строк, контент по ним достаёт рендер, поэтому отдельный `IDiffContentSource` из первого
наброска не понадобился.

Логика свёртки у upstream — `UnchangedRegion.fromDiffs` в
`editor/browser/widget/diffEditor/diffEditorViewModel.ts:458` (параметры `minimumLineCount` +
`contextLineCount`). Класс завязан на `observable`, но функция короткая: случай «прочитать и
написать своё», а не копировать.

### [x] Этап 1. Живой гуттер

**Проблема.** Change-bars в гуттере есть end-to-end, но залипшие: `extensions/git/main.ts`
пересчитывает ханки по `onDidSaveTextDocument`, поэтому во время набора полоски врут. В VS
Code они живут посимвольно.

**Что делаем.** Вычисление диффа переезжает из субпроцесса-расширения туда, где живёт буфер —
в основной процесс. Сервис держит текст `HEAD` для открытых файлов, по debounce'нутому
`onDidChangeContent` гоняет `DefaultLinesDiffComputer` и пушит `IGutterChangeDecoration`
в существующий шов. Контент `HEAD` берём у git-расширения приватной командой (двусторонние
команды уже работают) — полноценный content-provider тут ещё не нужен.

**Почему первым.** Валидирует вендоринг в рендере, а не в юнитах; чинит настоящий дефект;
не требует ни абстракции панели, ни diff view model; создаёт первую половину порта
`IDiffProvider` и вынуждает решить вопрос двух `IRange` (см. «Сквозные вопросы») на
маленькой задаче.

**Смотреть у upstream.** `contrib/scm/browser/quickDiffModel.ts` (508) — держит оригинал
моделью, `onDidChangeContent → triggerDiff`; `quickDiffDecorator.ts` (459).

**Наши файлы.** `extensions/git/main.ts`, `src/vs/workbench/api/common/iEditorDecorationsService.ts`,
`src/vs/workbench/browser/parts/editor/editorComponent.ts` (`setGutterChangeDecorations`),
новый сервис в `src/vs/workbench/contrib/`.

**Сделано** двумя PR: (1) реестр провайдеров ФС по схеме + `git:` в расширении
(`registerFileSystemProvider` раскомментирован по bounded member-level правилу, мост
host↔subprocess по образцу folding-провайдера); (2) `QuickDiffService` — дифф буфера против
версии из HEAD с debounce, пересчётом по сдвигу HEAD и по смене темы. Расширение потеряло
вычисление ханков, `extensions/git/lib/diff.ts` удалён.

**Что поймал реальный запуск (а тесты — нет).** SCM-расширение активируется асинхронно и
обычно уже ПОСЛЕ открытия первого файла, поэтому стартовый пересчёт видел «поставщика нет».
Этот отрицательный ответ кэшировался — бары не появлялись до первой правки. Починено двумя
вещами: реестр обзавёлся `onDidChangeProviders`, а сервис перестал кэшировать отрицательные
ответы. Плюс расширение регистрирует команду `provideOriginalResource` ДО провайдера схемы:
нотификации идут по каналу в порядке отправки, и к моменту пересчёта команда уже есть.
Регрессия закрыта тестом (снятие фикса красит его).

**Решение по геометрии зафиксировано:** адаптер в одной точке
(`quickDiffDecorations.ts`), миграцию ядра на upstream-примитивы не начинали.

**Сделано.** `src/vs/editor/common/diff/diffViewModel.ts` — `DiffViewModel` отдаёт
`rows: IDiffViewRow[]` (`unchanged` / `deleted` / `added` / `collapsed`) и `regions`, умеет
`expandRegion(index, from?, lineCount?)` и `expandAll()`. Алгоритм свёртки портирован с
`UnchangedRegion.fromDiffs`; инверсия изменений взята из уже перенесённого
`LineRangeMapping.inverse` (`rangeMapping.ts:21`). Номера строк на выходе 0-based, конверсия
из 1-based `LineRange` — в одном месте, как в `quickDiffDecorations.ts`.

Проверено на реальных данных: `extensionHost.ts` за 30 коммитов — 985 строк вью в

«видимые + скрытые = длина файла» держится в обоих режимах, набор добавленных строк
совпадает с `git diff`.

### [x] Этап 3. Абстракция editor pane

**Сделано.** `IEditorPane` (`browser/parts/editor/iEditorPane.ts`) — что группе нужно от
любой вкладки: `uri`, `view`, `isModified`, `onDidChangeState`, `focusEditor`. Бывший

панель не-текстового вида даёт вкладку рядом с файлом и рисует свой контент.

      одной. Чтобы открывать только дифф, нужен read рабочего дерева в браузерном слое
      (`file:`-провайдер / чтение буфера без открытого редактора) — тогда `modifiedText`
      берётся напрямую.
- [ ] **Живой дифф вместо снимка.** Вкладка держит тексты на момент открытия; правки в
      исходном буфере её не двигают. Нужна подписка на чужую вкладку и пересчёт по debounce —
      механика та же, что в `QuickDiffService`.
- [ ] **Раскрытие свёрнутых кусков жестом.** Модель это умеет
      (`DiffViewModel.expandRegion`), элемент — ещё нет: нужен клик/клавиша на строке-
      плейсхолдере. Просится в один шаг с горизонтальной прокруткой (`DiffViewElement.scrollLeft`
      сейчас константный ноль, длинные строки обрезаются) — оба жеста про «показать больше».
- [ ] **Сравнение произвольных ревизий и двух файлов.** Сейчас только «активный файл ↔ HEAD».

## Сквозные вопросы

Не этапы, а решения, которые придётся принять внутри этапов.

- [x] **Столкновение геометрий — решено на этапе 1.** Выбран адаптер в одной точке
      (`contrib/scm/browser/quickDiffDecorations.ts`): конверсия 1-based/`endExclusive`
      upstream ↔ 0-based/включительный `end.line` нашего гуттера, плюс граничная строка для
      удалений. Миграцию ядра на upstream-примитивы не начинали — несоразмерно ради одного
      места вызова; долг остаётся.
- [ ] **Приватные команды или живой `scm` (пункт F) — решается на этапе 6.** До Changes
      приватных команд к git-расширению достаточно. Канонический путь нужен, когда появится
      второй источник SCM-данных или сторонние расширения; тогда же всплывёт
      `registerTextDocumentContentProvider` и полноценный пункт **B**.
- [x] **Судьба `extensions/git/lib/diff.ts` — удалён на этапе 1.** Он парсил только заголовки
      ханков; после переезда гуттера на ядро стал мёртвым. Для диффа коммитов на этапе 6+
      понадобится другой парсер — с **телами** ханков, а не только с заголовками.
