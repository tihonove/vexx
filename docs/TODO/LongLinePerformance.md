# Производительность на очень длинных строках

**Статус: базовый фриз устранён.** Обнаружено при работе Output-панели: канал
`extensions.host.rpc` пишет строки в десятки-сотни тысяч символов, и на них
редактор ощутимо подвисал, подвешивая интерфейс целиком.

## Что сделано

Порог рендера `STOP_RENDERING_LINE_AFTER = 10 000` (`tuidom/common/textLimits.ts`),
аналог `editor.stopRenderingLineAfter` в VS Code (отдельная ручка от
токенизационного `MAX_LINE_LENGTH = 20 000`):

- **`DisplayLine` получил опциональный `stopAfter`** (`tuidom/common/displayLine.ts`):
  за порогом сегментируется только префикс, `columnMap` аллоцируется по длине
  префикса (не всей строки), выставляется `isTruncated`. Дефолт `Infinity` — не-
  редакторные вызовы (лейблы, truncation, tree, quickpick) не тронуты.
- **Инкрементальный кеш ширины** `LineWidthCache`
  (`src/vs/editor/common/viewModel/lineWidthCache.ts`) вместо пересчёта всего
  документа: калька `DocumentTokenStore`, подписан на `onDidChangeContent`,
  пересчитывает только изменённые строки безаллокационным `measureTextWidth`
  (`tuidom/common/measureTextWidth.ts`) с тем же порогом. `EditorElement.contentWidth`
  делегирует сюда. **Это и снимает Output-фриз** (append бампает `versionId`, но
  пересчитываются только 1–2 строки, а не весь документ).
- **Единый seam** `EditorViewState.displayLineFor()` с порогом — через него идут
  все per-строчные постройки `DisplayLine` в редакторе (рендер, каретка,
  навигация по словам, hit-test), поэтому длинная строка перестала быть O(длины)
  на любом пути.
- **Маркер обрезки** `…` в рендере на месте отреза (как overflow-эллипсис в VS Code).

Замеры после фикса (документ 200 строк + одна 200k-строка): 200 Output-append'ов с
чтением `contentWidth` — ~5 мс (было ~14 с); первый `contentWidth` на открытии —
~2 мс (было ~72 мс). Демо — `e2e/scenarios/longLine.scenario.ts`.

## Осознанно НЕ сделано (следующие задачи)

- **reveal-по-клику** за порогом (VS Code `longLinesHelper` снимает cap на клик за
  местом отреза) — сейчас курсор за порогом клампится к маркеру.
- **Конфиг** `editor.stopRenderingLineAfter` — порог захардкожен константой.
- **Ленивое окно вьюпорта**: даже с cap `DisplayLine` аллоцирует до 10k слотов, а
  `charAtColumn`/`graphemeAtColumn` — линейный скан по слотам; окно по видимым
  колонкам убрало бы и это. Тогда же — горизонтальный скролл в хвост без обрезки.
- **large-file режим** (VS Code `largeFileOptimizations`): порог по размеру файла,
  гасящий дорогие фичи при открытии гигантских файлов.

## Исходный анализ (для истории)

## Замеры

Документ = 200 обычных строк + одна длинная; вьюпорт 90×17:

| Длина строки | `new DisplayLine` | `contentWidth` | первый render | повторный render |
|---|---|---|---|---|
| 1 000 | 2.1 ms | 3.8 ms | 8.0 ms | 2.4 ms |
| 10 000 | 4.9 ms | 6.3 ms | 2.5 ms | 2.2 ms |
| 100 000 | 28.2 ms | 46.9 ms | 2.6 ms | 1.9 ms |
| 200 000 | 72.1 ms | 72.1 ms | 2.3 ms | 1.9 ms |

**Сама отрисовка дешёвая и от длины не зависит** — рисуются только видимые
колонки. Всё время уходит в разбор строки целиком.

## Причина

`DisplayLine` (`tuidom/common/displayLine.ts`) прогоняет `Intl.Segmenter` по
**всей** строке и аллоцирует `GraphemeSlot` на каждый графемный кластер —
линейно по длине, независимо от того, сколько колонок видно.

Дальше это умножается на два обстоятельства:

1. `EditorElement.contentWidth` (нужен горизонтальному скроллбару) строит
   `DisplayLine` для **каждой** строки документа. Кеш есть, но по `versionId`
   документа — то есть сбрасывается на любой правке.
2. В Output каждая входящая запись бампает `versionId`. При активном RPC-трейсинге
   это означает полный пересчёт (~72 ms при одной 200k-строке) **на каждую
   строку лога**.

## Куда копать

- **Пер-строчный кеш ширины** вместо пересчёта всего документа: при append
  меняется только последняя строка, остальные ширины уже известны.
- **Лимит на разбор длинной строки** — аналог `editor.maxTokenizationLineLength`
  в VS Code (20 000 по умолчанию): за порогом отключать «дорогие» фичи и считать
  ширину приближённо.
- **Ленивый `DisplayLine`** — считать слоты только для запрошенного диапазона
  колонок, а не для всей строки.
- VS Code поверх этого ещё и сворачивает длинные строки с кнопкой разворачивания —
  отдельная фича, но она же снимает проблему с глаз пользователя.
