### BUG-2 — гонка активации: команда сообщает «нет версии в git» для закоммиченного файла

**Репро.** На чистом файле сразу после старта (первый прогон) вызвать
`Compare Active File with HEAD` **до** того, как git-расширение зарегистрировало
`git:`-провайдер ФС. Появляется статус-бар «**No changes to compare: the file has no
version in git**». Для закоммиченного файла это неверно — версия в git есть, просто
провайдер ещё не поднялся. Воспроизводится недетерминированно (окно гонки — между
активацией расширения и первым вызовом команды).

**Причина.** `readOriginal` (`compareWithHeadAction.ts`) для трёх разных ситуаций —
untracked-файл, git недоступен, провайдер ещё не активировался — возвращает `null`, и
команда показывает одно и то же сообщение. В отличие от живого гуттера, у команды нет ни
ретрая, ни подписки на `onDidChangeProviders`: разовый вызов ловит тот момент, когда
провайдера ещё нет, и трактует его как «сравнивать не с чем».

Тот же класс гонки, что чинили для гуттер-баров на этапе 1 (см. [Diff.md](Diff.md),
«Что поймал реальный запуск»), но там лечили реестром (`onDidChangeProviders` + отказ
кэшировать отрицательные ответы), а команда Compare этого не использует.

**Варианты фикса** (направление не выбрано):
- дождаться готовности провайдера — например, при `provideOriginalResource != null`, но
  `!hasProvider(scheme)`, подождать `onDidChangeProviders` с коротким таймаутом и повторить
  чтение (ближе всего к тому, как решён гуттер);
- либо разделить сообщения: пока провайдер не поднялся, показывать «git ещё
  инициализируется», а «нет версии в git» — только когда провайдер есть, а оригинала нет;
- либо, минимально, смягчить формулировку до «no git version available yet», чтобы она не
  утверждала факт, которого команда в этот момент знать не может.

**Severity — низкий:** окно узкое (только сразу после старта, до первой активации), данные
не теряются и не портятся, повтор вызова через мгновение уже работает. Но пользователь
получает уверенно-ложное сообщение о своём файле.
