# Нижняя Panel и терминал — баги фокуса и жизненного цикла

Дефекты, найденные при e2e-тестировании MVP «панель консоли» (нижняя Panel + вкладка
TERMINAL). Общий корень у первых трёх — **никто не владеет фокусом при уходе виджета
терминала со сцены**: панель прячется или шелл умирает, а фокус остаётся на элементе,
которого на экране нет.

Фича и её архитектура — [IntegratedTerminal.md](IntegratedTerminal.md); эта задача только про
исправления. Все пункты воспроизведены на настоящем бинаре (headless) — сьют
`e2e/console-panel.test.ts`, тесты с префиксом `BUG-*` красные намеренно и станут
регрессионными после фикса.

## Задачи

- [x] **BUG-1 — ввод при скрытой панели уходит в невидимый шелл и выполняется** (критично)
- [x] **BUG-2 — после скрытия панели клавиатура не возвращается в редактор** (высокий) —
  закрыт тем же фиксом, что BUG-1: у них общий корень
- [ ] **BUG-3 — после выхода из шелла ввод пропадает полностью** (высокий)
- [ ] **BUG-4 — Toggle Terminal после смерти шелла прячет панель вместо нового шелла** (средний)
- [x] **BUG-5 — активная вкладка панели не переживает рестарт** (низкий)
- [x] **BUG-6 — колесо мыши не скроллит вывод терминала** (средний, по сути отсутствующая фича)
- [ ] **Enabler — `TUIDom.sendMouse` в инспектор-протоколе**, без него мышь e2e непокрываема

---

### BUG-1 / BUG-2 — фокус остаётся на скрытом терминале

**Репро.** Toggle Terminal → терминал в фокусе → `Ctrl+J` (панель скрыта) → печатаем текст.
Текст не появляется в редакторе; при повторном `Ctrl+J` он обнаруживается в шелле.
С Enter **команда реально выполняется**: в тесте `touch <файл>` вслепую создал файл.

**Почему.** Видимость панели меняется по цепочке `LayoutService.setPanelVisible` →
`PanelService.setVisible` → подписчик двигает `WorkbenchLayoutElement` и ставит контекст-ключ
`panelVisible`. Фокуса не касается никто: ни `panelService.ts`, ни `layoutService.ts`, ни
`terminalPanelComponent.ts` (там есть только `focus()` на `onDidRequestFocus`).
`PanelContainerElement.getChildren()` перестаёт отдавать виджет, но `TerminalViewElement`
остаётся `activeElement` у FocusManager и продолжает жрать клавиатуру.

**Что делать.** При скрытии панели (и при уходе с вкладки, чей контент держит фокус)
возвращать фокус предыдущему владельцу — как VS Code возвращает его в редактор. Разумное
место — владелец layout'а (`WorkbenchComponent`) или сам `PanelComponent`: подписка на
`onDidChangeVisibility`/`onDidChangeActiveView`, и если текущий `activeElement` лежит внутри
поддерева панели — отдать фокус редактору.

**Сделано.** `PanelFocusContribution` (`browser/parts/panel/panelFocusContribution.ts`):
подписка на `onDidChangeVisibility`/`onDidChangeActiveView`, и если `activeElement` лежит
внутри поддерева `PanelComponent.view` — фокус уходит в редактор (а если редакторов нет,
просто снимается). Регрессия — `panelFocusContribution.test.ts` поверх app-харнесса.

### BUG-3 — после `exit` ввод исчезает

**Репро.** Открыть терминал → `exit` → вкладка показывает «No active terminal.» → печатать.
Текст не появляется ни в редакторе, ни в панели. Команды (Ctrl+J) при этом работают —
диспетчер клавиш висит на body.

**Почему.** `terminalPanelComponent.handleClose` диспозит виджет и ставит контент вкладки в
`null`, но фокус остаётся на снятом с дерева элементе. Тот же корень, что BUG-1/2, только
триггер — смерть сессии, а не скрытие панели.

**Что делать.** Тем же механизмом: если фокус был на закрываемом виджете — вернуть его
редактору (или следующему терминалу, если он есть).

### BUG-4 — Toggle Terminal после смерти шелла только прячет панель

**Репро.** `exit` → панель осталась открытой с placeholder'ом → Toggle Terminal → панель
просто скрылась. Чтобы получить новый шелл, команду надо позвать дважды.

**Почему.** `terminalActions.ts` (`toggleTerminalAction`): условие
`showing = layout.isPanelVisible() && panel.getActiveViewId() === TERMINAL_VIEW_ID`
не смотрит, есть ли живые инстансы.

**Что делать.** Добавить в условие `&& terminalService.hasOpenTerminals` — тогда при пустой
вкладке команда пойдёт по ветке «показать + заспавнить».

### BUG-5 — активная вкладка панели не переживает рестарт

**Репро.** Оставить открытой вкладку TERMINAL → перезапуск → панель восстановилась видимой,
но активна PROBLEMS.

**Почему.** В `src/vs/workbench/common/stateKeys.ts` есть только `PANEL_VISIBLE_STATE` и
`PANEL_HEIGHT_STATE`; активная вкладка не персистится, а `PanelService` делает активной
первую зарегистрированную (PROBLEMS).

**Что делать.** Ключ `PANEL_ACTIVE_VIEW_STATE` + восстановление в `layoutService.restoreLayout`
рядом с видимостью/высотой. Терминал при этом сознательно **не** респавним — вкладка
восстанавливается с placeholder'ом (ленивый спавн остаётся ленивым).

### BUG-6 — колесо мыши не скроллит вывод терминала

**Репро.** Вывести в шелле больше строк, чем помещается (`seq 1 100`), крутить колесо — ничего.
Внутри программ, которые сами включают mouse-tracking (`less --mouse`, `htop`, `vim`),
колесо работает.

**Почему.** Две независимые причины:
1. `terminalViewElement.ts` отдаёт wheel **только** в `surface.sendMouse` →
   `coreMouseService.triggerMouseEvent`, а тот репортит событие программе лишь если она
   включила mouse-режим. Bash не включает → событие выброшено.
2. Даже если бы дошло — рисовать нечего: `embeddedTerminalSession.readCell` всегда читает
   `buffer.baseY + y`, то есть дно живого вьюпорта. В `ITerminalSurface` нет ни смещения
   прокрутки, ни понятия «смотрим в скролбэк». Скролбэк копится (`scrollback: 1000`) и
   недостижим; клавиатурного скролла (Shift+PageUp) тоже нет.

**Что делать** (как в xterm.js/VS Code): завести во `ITerminalSurface` состояние вьюпорта —
`scrollLines(delta)` + чтение `readCell` со смещением (`baseY - scrollOffset + y`); в виджете
обрабатывать wheel **локально**, когда программа mouse-tracking не включила (у xterm для этого
есть `coreMouseService.areMouseEventsActive`); сбрасывать смещение на ввод и на новый вывод;
прятать курсор, пока смотрим в скролбэк. Дальше на этом же вьюпорте вырастают выделение и
копирование из «UX шелла» в [IntegratedTerminal.md](IntegratedTerminal.md).

**Сделано.** Во `ITerminalSurface` появилось состояние вьюпорта (`scrollOffset`, `scrollLines`,
`mouseEventsActive`); `readCell` читает строку со смещением, курсор в скролбэке прячется, ввод и
новый вывод возвращают вьюпорт на дно. Виджет крутит вьюпорт сам, пока mouse-tracking выключен,
и понимает Shift+PageUp/PageDown. Регрессия закрыта юнитами (`terminalViewElement.scroll.test.ts`,
`embeddedTerminalSession.scroll.test.ts`); e2e по-прежнему ждёт enabler `TUIDom.sendMouse`.

### Enabler — `TUIDom.sendMouse` в инспектор-протоколе

Сейчас протокол умеет `sendKey/sendText/resize/captureFrame` (`tuidom/inspector/protocol.ts`),
инъекции мыши нет — поэтому e2e не покрывает **клик по табам панели, драг сэша (высота панели),
скролл колесом и мышь внутри шелла**; всё это проверяется только руками. Добавление
`TUIDom.sendMouse` (координаты + кнопка + действие + модификаторы) закрывает разом и регрессию
на BUG-6, и клики по вкладкам. См. [Inspector.md](Inspector.md), [E2E.md](E2E.md).

## Что при тестировании оказалось в порядке

Ctrl+J показывает панель с обеими вкладками · Toggle Terminal поднимает живой шелл ·
Ctrl+C прерывает выполняющуюся команду · ресайз окна рефлоуит шелл, он остаётся рабочим ·
схлопывание окна до 40×6 не роняет приложение · содержимое терминала переживает hide/show ·
PTY не остаётся сиротой после выхода из приложения.

Непокрытым остаётся всё мышиное (см. enabler) плюс тема/ANSI-палитра, wide-chars, живой
Ctrl+`` ` `` на tier kitty/csi-u и вся кросс-платформенность (macOS/Windows).
