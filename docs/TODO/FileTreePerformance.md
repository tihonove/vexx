# [~] Производительность больших файловых деревьев

При открытии большой директории Vexx тормозит: медленное открытие воркспейса и
лаги при навигации по дереву файлов. Эта задача — **диагностика** узких мест и
**фиксация baseline бенчмарками**. Сами фиксы — отдельным этапом (см. ниже).

## Как запускать бенчи

```
npm run test:perf
```

Конфиг — `vitest.perf.config.ts` (отдельный от обычного `npm test`, не влияет на
coverage). Бенчи: `src/**/*.bench.ts`, общие фикстуры — `src/TestUtils/perfFixtures.ts`.

| Бенч | Что меряет |
| --- | --- |
| `FileSearchService.bench.ts` → `activate / index N files` | время фонового обхода до готовности индекса (с уступками event loop) |
| `FileSearchService.bench.ts` → `search …` | стоимость одного запроса в Quick Open |
| `FileTreeDataProvider.bench.ts` → `getChildren …` | раскрытие одного большого каталога |
| `TreeViewElement.bench.ts` → `refresh / toggleExpand / render` | flatten + O(N) сканы + рендер кадра |

## Baseline (измерено, dev-машина — порядок величин, не абсолют)

| Бенч | mean |
| --- | --- |
| `activate / index 1000 files` | ~5 мс |
| `activate / index 10000 files` | ~50 мс (линейно блокирует старт) |
| `search` пустой запрос | ~0.001 мс |
| `search` basename-фрагмент (10k индекс) | ~7.6 мс на нажатие |
| `search` path-фрагмент (10k индекс) | ~12.5 мс на нажатие |
| `getChildren` 5000 записей | ~7.9 мс |
| `refresh()` дерева на 2000 раскрытых узлов | ~35 мс |
| `render viewport` (40 строк) | ~0.85 мс/кадр (норм) |

Чтения: индексация линейна по числу файлов и блокирует старт; `search` — заметные
миллисекунды на каждое нажатие уже на 10k; render кадра ограничен вьюпортом и
дёшев. Числа — для отслеживания регрессов, абсолют зависит от машины.

## Диагностика (узкие места)

Обе подсистемы стартуют в `AppController.setWorkspaceFolder()`, который зовётся в
`main.ts` **до** `app.run()` — т.е. блокирует первый кадр.

### Аспект 1 — построение поискового индекса (главный блокер)
`src/Controllers/FileSearchService.ts`:
- `walkSync`/`walkDir` — **синхронный** рекурсивный `fs.readdirSync` по всему
  дереву на main-треде. O(N файлов), блокирует event loop до завершения.
- `chokidar.watch(root, { depth: undefined })` — **рекурсивный watcher на весь
  воркспейс**. Второй крупный стартовый расход + постоянная нагрузка CPU/память;
  вероятный источник лагов при навигации (поток fs-событий).
- `search()` — линейный O(N) скан с fuzzy-match на каждое нажатие.

### Аспект 2 — TreeView
Загрузка ленивая (один уровень на раскрытие), рендер ограничен вьюпортом — это ОК.
Ограниченные, но реальные расходы:
- `refresh()` → `findElementByKey` и `restoreSelection` — линейные O(flatNodes)
  сканы; refresh летит по 300ms-debounce на каждое изменение каталога.
- `renderViewport` аллоцирует `new DisplayLine(rowText)` на каждую видимую строку
  каждый кадр.
- `FileTreeDataProvider.readDirectory` — `readdirSync` + sort + `getFileIcon` на
  каждый элемент при раскрытии большого каталога.

**Вывод:** доминирует Аспект 1 (синхронный walk + рекурсивный watch при открытии).

## Сделано — Аспект 1 переведён в фон

`FileSearchService` переписан: индекс строится **в фоне** чанкованным async-обходом
(`fs.promises.readdir`, уступка event loop через `setImmediate` после каждой
директории). `activate()` неблокирующий, возвращает `ready`. **Рекурсивный
chokidar-watcher на весь воркспейс убран** — он и душил event loop (отсюда лаги
всего редактора, даже без поиска). Свежесть теперь по триггеру: `refreshIfStale()`
при открытии Quick Open (троттлинг), результаты в пикере наполняются вживую через
`onIndexChanged`. `search()` работает по частичному индексу во время обхода.
Допускается eventual freshness (только что созданный файл найдётся не сразу).

> NB про бенч `activate`: метрика сменила смысл. Раньше — время блокировки
> main-треда; теперь — суммарное время фонового обхода **с** уступками event loop
> (поэтому абсолют чуть выше: ~66 мс на 10k против ~50 мс синхронных). Главный
> выигрыш — event loop больше не заблокирован — этим числом не ловится; его видно
> вручную (редактор интерактивен во время индексации).

## Сделано — ускорение `search` (precompute + debounce)

Шаг 1 оптимизации самого поиска (движок не меняли):

- **Предвычисление кейс-фолдинга.** `FileSearchEntry` теперь несёт готовые
  `basename` / `basenameLower` / `relativePathLower`, посчитанные **один раз** при
  обходе. Раньше `search()` на каждое нажатие звал `path.posix.basename()` (N
  substring-аллокаций) и `fuzzyMatchBest`, который размножал `toLowerCase(text)` до
  ~9× и `toLowerCase(query)` до ~8× **на запись**. Теперь `query` лоукейсится один
  раз на поиск, а движок `FuzzySearch` получил `fuzzyMatchLower`/`fuzzyMatchBestLower`
  — варианты, принимающие уже-готовые строки (публичные `fuzzyMatch`/`fuzzyMatchBest`
  остались тонкими обёртками, поведение идентично). Память: +~60 байт/файл (~0.6 МБ
  на 10k) — приемлемо.
- **Дебаунс поиска (ведущий фронт).** `QuickOpenController` в file-режиме не гоняет
  O(N) поиск на каждый символ: одиночное нажатие в простое выполняется синхронно
  (leading edge), серия быстрых нажатий коалесцируется в один хвостовой прогон
  (`SEARCH_DEBOUNCE_MS = 16`). Command-режим и наполнение по `onIndexChanged` —
  по-прежнему синхронны.

Замер (`npm run test:perf`, dev-машина, порядок величин):

| Бенч | было | стало |
| --- | --- | --- |
| `search` basename-фрагмент (10k) | ~7.6 мс | **~3.7 мс** (~2×) |
| `search` path-фрагмент (10k) | ~12.5 мс | **~7.9 мс** (~1.6×) |

Дебаунс вдобавок снижает **частоту** прогонов при быстрой печати — этим числом не
ловится, видно вручную (плавная печать в Quick Open на большом дереве).

## Сделано — отсев по bitmask (строим заранее)

Шаг 2 ускорения `search`: дешёвый **отсев присутствия символов** до дорогого
fuzzy-match. Движок не меняли.

- **Маска символов, посчитанная заранее.** В `makeEntry()` (во время обхода) для
  каждой записи считаем 32-битную `charMask` (бит `code & 31` на каждый символ)
  по `basenameLower` и `relativePathLower` → поля `basenameBits` / `relativePathBits`
  (+~16 байт/файл). `charMask` — в `src/Common/FuzzySearch.ts`.
- **Необходимое условие в `search()`.** Маску запроса считаем один раз; в горячем
  цикле `(entry.relativePathBits & queryBits) !== queryBits` отбрасывает запись
  одним целочисленным AND, **не вызывая** `fuzzyMatchBestLower`. Путь содержит
  basename, поэтому маска пути — глобальное необходимое условие для любого
  совпадения; маска basename дополнительно решает, пробовать ли basename-матч.
- **Корректность.** Это необходимое условие fuzzy-матча (все символы запроса
  обязаны присутствовать), поэтому отсев **никогда** не выбрасывает настоящее
  совпадение. Коллизии символов в 32 бита допустимы (лишь чуть снижают
  селективность). Состав и порядок выдачи не меняются — все прежние тесты зелёные,
  плюс регресс-тест «выдача с отсевом == сырой матчер» в
  `FileSearchService.Search.test.ts`.

Замер (`npm run test:perf`, та же машина, до — `main`, после — эта ветка):

| Бенч (10k) | до (main) | после | выигрыш |
| --- | --- | --- | --- |
| `search` basename-фрагмент | ~3.6 мс | **~1.9 мс** | ~1.9× |
| `search` path-фрагмент | ~7.1 мс | **~5.4 мс** | ~1.3× |
| `search` непроходной запрос | ~1.6 мс | **~0.03 мс** | **~59×** |

Главный выигрыш — на **селективных/непроходных** запросах (типичный случай во
время набора: большинство записей не содержат всех символов и отсекаются мгновенно).
