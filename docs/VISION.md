# Vision — позиционирование Vexx

Этот документ — не про архитектуру и не про фичи. Он про то, **чем Vexx является для внешнего
мира** и куда осмысленно двигаться. Цели и неизменяемые ограничения реализации — в
[GOAL.md](../GOAL.md); здесь — стратегия, которая пересматривается по мере того, как меняются
факты.

Документ живой. Если факт из раздела «Ограничения реальности» перестал быть фактом — направление
пересматривается.

---

## Активы — что у нас есть на самом деле

Ранжировано **по редкости**, а не по вложенному труду. Это важно: самое ценное здесь — не то, что
выглядит как продукт.

1. **VS Code-совместимый extension host вне Electron и браузера.** Расширения VS Code, запущенные
   там, где нет ни Electron, ни браузера. Аналогов, по сути, нет. Переиспользуемо далеко за
   пределами нашего редактора — вплоть до выдачи языковых фич расширений наружу как инструментов.
2. **TUI-движок на TypeScript, который не Ink.** Grid + double buffering + диффы ANSI + layout +
   Kitty keyboard protocol. Node-based AI CLI сегодня рендерят терминал через Ink/React и страдают;
   у нас лучше по технике и в том же языке.
3. **LSP.** Работает — значит это IDE, а не «редактор с подсветкой».
4. **Собственно редактор.** Самый слабый актив: конкуренты (Neovim, Helix) бесплатны и превосходны.

**Вывод, который надо держать в голове:** то, что мы считаем продуктом (4), стоит меньше всего.
То, что мы считаем инфраструктурой под продукт (1–2), — единственное, чего рынок реально хочет.

## Ограничения реальности

Потолки, которые нам не сдвинуть. Стратегия обязана их обходить, а не бороться с ними.

- **Marketplace VS Code закрыт юридически.** ToS Microsoft запрещает сторонним продуктам ходить в
  marketplace — отсюда план с GitHub-топиком и OpenVSX (см. [TODO/Extensions.md](TODO/Extensions.md),
  Phase 10). Следствие жёстче, чем кажется: на OpenVSX нет ровно того, ради чего терпят VS Code —
  Pylance, C#, Copilot, Remote-SSH. Они лицензированы только для VS Code.
- **Webview'ы в TUI невозможны в принципе.** На них сидит огромная доля расширений. Потолок
  совместимости — языковые и тулинговые расширения, не UI-heavy.
- **Функциональный паритет с VS Code недостижим.** Это не горизонт, это асимптота, часть которой
  закрыта by design (два пункта выше). Строить позиционирование на паритете нельзя.
- **Редакторы не монетизируются.** VS Code бесплатен, потому что Microsoft продаёт Azure и GitHub.
  Neovim и Helix бесплатны. Sublime — единственный работающий коммерческий редактор, это маленький
  бизнес, который строили 15 лет. Деньги в редакторах 2026 — это AI-наценка, а редактор канал
  дистрибуции. Продавать Vexx нельзя ни в каком виде.
- **`ide`-протокол Claude Code закрыт.** Официальные интеграции VS Code и JetBrains работают через
  встроенный MCP-сервер `ide`: WebSocket на loopback, случайный порт, токен в
  `~/.claude/ide/<port>.lock`. Протокол **не документирован и не открыт третьим сторонам**. Реверс
  возможен, но это внутренний RPC — ломается на любом релизе.
- **Авторизация claude.ai запрещена третьим лицам.** Дословно из доков Agent SDK: *«Unless
  previously approved, Anthropic does not allow third party developers to offer claude.ai login or
  rate limits for their products, including agents built on the Claude Agent SDK.»* Значит
  **встроенный в Vexx агент потребует от пользователя отдельный `ANTHROPIC_API_KEY` и отдельный
  чек** — при том что у него уже есть Max, через который он гоняет Claude Code каждый день. Для
  бесплатного редактора с ровно этой аудиторией — это стена, а не трение.

### Что при этом можно

- **Стрим Claude Code доступен.** `claude -p --output-format stream-json` отдаёт NDJSON:
  `system/init`, вызовы и результаты инструментов, `api_retry`, финальный `result`. С
  `--include-partial-messages` — токенные дельты. Всё документировано.
- **Agent SDK (TS) спавнит тот же CLI** и говорит с ним JSON по stdio — он не переписывает харнесс
  и не ходит в API напрямую. «SDK или сырой stream-json» — выбор тактический, не стратегический.
  Но оба упираются в проблему авторизации выше.
- **Дослать ход в уже запущенный `claude -p` нельзя** (одна инвокация — один промпт, дальше
  `--resume <session-id>`). **Подключиться наблюдателем к чужой интерактивной сессии нечем.**
  Транскрипты в `~/.claude/projects/**/*.jsonl` лежат, но доки прямо предупреждают: формат
  внутренний, меняется между релизами, не парсить.
- **MCP наружу и хуки — документированы и поддерживаются.** Claude Code штатно ходит в сторонние
  MCP-серверы; `PreToolUse`-хуки штатно перехватывают вызовы инструментов. Это наша легальная
  дверь.

## Анти-цели

Чего мы сознательно **не** делаем:

- **Не позиционируемся как клон VS Code.** В этой рамке мы всегда хуже: любой тред заканчивается
  «а зачем, есть Remote-SSH / code-server». Клоны не получают внимания, они получают сравнения.
- **Не гонимся за паритетом фич.** См. ограничения.
- **Не монетизируем редактор.** Ни лицензия, ни open core, ни pro-фичи.
- **Не реверсим закрытые протоколы.** `ide` MCP Claude Code — мимо: фрагильно, неподдерживаемо, и
  ставит нас в позу самозванца вместо соседа.
- **Не строим стратегию на своей трактовке чужого ToS.** Серая зона (пользователь сам залогинен, мы
  просто спавним его `claude`) читается неоднозначно — а цель присоседиться, а не поругаться.

## Направления

Пять вариантов. Не взаимоисключающие — но у них разная цена и разный потолок.

### A. Reach — «VS Code там, куда он не дотягивается»

**Тезис:** не конкурент VS Code, а его продолжение туда, где GUI нет.

**Аудитория:** SRE, DevOps, embedded, HPC, работа по SSH, bastion-хосты, контейнеры, tmux, узкий
канал, залоченные окружения без браузера.

**За:** аудитория реальная, история честная, конфликта с VS Code нет по построению.
**Против:** эти люди не платят и почти не шумят. Потолок — звёзды на GitHub.
**Роль:** источник аудитории, не источник смысла.

### B. Агентный редактор — человек и агент в одном буфере

**Тезис:** агент правит код, ты видишь правки инлайн в редакторе и правишь дифф руками до апрува —
всё в терминале, по SSH, без GUI.

**За:** это опыт VS Code + Copilot, которого в терминале нет ни у кого. Отличный демо-кадр.
**Против:** если делать через встраивание Agent SDK — упираемся в стену авторизации (отдельный
API-ключ, второй чек). Плюс переизобретение UX-части харнесса и вечная гонка за их скоростью.
**Как обойти:** не встраивать агента. Пользователь запускает **свой** `claude` со своей подпиской, а
интеграция идёт через документированные поверхности (см. C).

### C. Extension host как инфраструктура — языковой мозг для любого агента ⭐

**Тезис:** Vexx отдаёт наружу MCP-сервер, который даёт **любому** агенту настоящие языковые фичи
VS Code — go-to-definition, find-all-references, rename-symbol, code actions, диагностики — вместо
grep'а и чтения файлов целиком.

**Механика (всё из легальных документированных кубиков):**
- **MCP-сервер поверх extension host + LSP (pull).** Claude Code штатно ходит в сторонние
  MCP-серверы. Инструменты: языковая интеллектуальность, активный файл, выделение, открыть дифф.
- **`PreToolUse`-хуки (push).** Перехват Edit/Write → правка едет в наш diff-вьювер, а не текстом в
  консоль.
- Авторизация — пользователя, его подписка, его `claude`. Ноль ToS-проблем, ноль второго чека.

**За:** функционально это **то же, что делает официальный `ide`-сервер** для VS Code и JetBrains —
но легально и без реверса. Плюс то, чего у официального нет и не будет: их `ide` отдаёт диагностики
и умеет запускать питон в ноутбуке, наш отдаёт весь языковой мозг VS Code. Работает с любым агентом,
не только с Claude. Предъявляется **отдельно от редактора** — редактор становится витриной, а не
просителем.
**Против:** сторонний MCP-сервер вызывается **по решению модели** — привилегий официального `ide` у
нас не будет (Claude Code не станет сам спрашивать наше выделение по `@`). Хуки закрывают часть
разрыва, но не весь. Это цена легального пути.

### D. TUI-движок как компонент

**Тезис:** TUIDom как замена Ink для Node-based CLI.

**За:** технически мы лучше, и в правильном языке.
**Против:** рынок крошечный, Ink бесплатен, а переезд на чужой движок — переписывание всего UI. Claude
Code закрыт, Gemini CLI на Ink, opencode/Crush на Go.
**Роль:** побочный эффект, не направление.

### E. Карьерный рычаг

**Тезис:** проект как proof-of-work. 72k строк, движок рендера с нуля, extension host на RPC, LSP —
соло за месяцы. Внимание конвертируется не в выручку, а в позицию.

**За:** это **самый реалистичный способ «чтобы платили за работу над редактором»** — как зарплата, а
не как revenue. Кандидаты: Coder, Gitpod, Warp, Replit, Sourcegraph, Zed, Anthropic/OpenAI/Cursor.
**Против:** не контролируется напрямую, зависит от момента и удачи.
**Роль:** способ получить деньги. Именно он, а не A–D.

## Текущая ставка

**C — ядро. B — витрина. A — аудитория. E — способ получить деньги. D — побочный эффект.**

Формулировка вовне: **не «VS Code в терминале», а «экосистема расширений VS Code без Electron —
и языковой мозг для терминальных агентов».** Клон приглашает сравнение; инфраструктура приглашает
разговор.

Ключевая мысль: обёртки живут до следующего релиза платформы. Экосистему расширений VS Code без
Electron Anthropic не построит никогда — это не их бизнес. Это наша постоянная территория.

## Демо, ради которого всё

Один кадр, который не показывает никто:

> Агент правит код по SSH в контейнере без GUI. Правки прилетают в редактор через `PreToolUse`-хук
> и ложатся инлайн. Ты правишь дифф руками до апрува. Агент делает rename через **настоящий language
> server**, а не регекспом — потому что мы дали ему свой extension host как MCP. Пользователь платит
> только за свою подписку.

Ни VS Code (нужен GUI), ни Claude Code в голом терминале (нет редактора и нет языкового мозга), ни
Neovim (нет extension host'а) этого не показывают.

**Релиз имеет смысл привязывать к этому кадру, а не к паритету фич.** Он стоит больше, чем все
остальные фичи вместе.

## Открытые вопросы

- Где живёт MCP-сервер относительно слоёв (Extensions / Controllers)? Какой минимальный набор
  инструментов делает демо неотразимым?
- Насколько далеко хуки закрывают разрыв с привилегиями официального `ide`-сервера?
- Стоит ли вообще держать встроенного агента как опцию для тех, у кого есть API-ключ, — или это
  распыление?
- Честная формулировка потолка совместимости («языковые и тулинговые расширения — да, webview —
  никогда») как публичная позиция: это сила, а не слабость.
