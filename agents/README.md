# @vexx/agents

Машинерия агентов Vexx. Отдельный пакет со своим `package.json` — свои зависимости,
свой `npm install`, никакого влияния на корневой проект.

Два кирпича:

1. **Декларативные состояния.** Набор лейблов `agent:*` и колонок доски описан в
   [`config.jsonc`](config.jsonc) и применяется командой `sync`. Добавили состояние в один
   файл → запустили sync → оно появилось в GitHub. Руками в веб-морде ничего не заводим:
   иначе конфиг перестаёт описывать реальность.
2. **Демон-оркестратор.** Один процесс держит тикер, MCP-сервер и витрину. Раз в N минут
   запускает свежий `claude -p /orchestrate`, который смотрит на мир и раздаёт задачи агентам.
   Поднимается корневым `./agents.sh` — см. раздел «Демон» ниже.

## Установка

```sh
cd agents && npm install
```

Синк полей проекта требует scope `project`, которого нет в дефолтном наборе `gh`.
Один раз, интерактивно:

```sh
gh auth refresh -s project
```

Без него `sync --labels-only` работает, а полный `sync` падает с кодом 2 и этой подсказкой.

## Команды

```sh
npm run sync:dry                 # показать план, ничего не менять
npm run sync                     # применить (аддитивно)
npm run sync -- --labels-only    # только лейблы репозитория
npm run sync -- --project-only   # только поля проекта
npm run sync -- --prune --yes    # + удалить лишнее
npm run typecheck && npm test
```

## Модель удаления

По умолчанию синк **аддитивен**: создаёт недостающее и правит разошедшееся, но ничего
не удаляет. `--prune` объявляет конфиг полным владельцем, и тогда:

- **лейблы** — удаляются только те, что начинаются с `labels.prefix` (`agent:`).
  Чужие (`bug`, `tech-debt`, `claude`, …) вне юрисдикции синка и не трогаются никогда;
- **опции полей** — удаляются те, которых нет в конфиге. Значение такой опции
  у карточек обнуляется, поэтому любое удаление дополнительно требует `--yes`;
- **поля** — не удаляются никогда. У них нет префикса-юрисдикции, встроенные
  (Title, Assignees, …) удалить нельзя в принципе, а удаление кастомного унесло бы
  данные всех карточек. Если тип поля в проекте разошёлся с конфигом — синк падает
  с объяснением, а не пересоздаёт поле.

## Как это устроено

- `src/gh.ts` — единственный канал к GitHub: обёртки над `gh` CLI и `gh api graphql`.
  Своих токенов у машинерии нет.
- `src/config.ts` — типы и валидация `config.jsonc`.
- `src/labels.ts`, `src/projectFields.ts` — чтение состояния, **чистые планировщики**
  (`planLabels`, `planFields`) и применение. Тесты бьют по планировщикам, поэтому
  идут без сети.
- `src/cli.ts` — разбор флагов, печать диффа, exit-коды: 0 — успех, 1 — ошибка
  применения, 2 — нет scope или невалидный конфиг.

Лейблы правятся через `gh label`, поля проекта — через GraphQL: `gh` не умеет
редактировать опции существующего single-select поля.

Важная деталь API: `updateProjectV2Field.singleSelectOptions` — **полная замена**
набора опций. Опция, переданная с `id`, сохраняется вместе с назначениями у карточек;
без `id` — создаётся; не переданная — удаляется. Поэтому в аддитивном режиме
незнакомые опции дописываются в хвост payload'а — иначе «ничего не удаляем»
превратилось бы в тихое удаление. Порядок опций в массиве = порядок колонок на доске.

# Демон

```sh
./agents.sh start      # поднять в tmux-сессии "agents" (идемпотентно)
./agents.sh status     # демон + живые агенты
./agents.sh tick       # тик прямо сейчас (то же, что кнопка на витрине)
./agents.sh attach     # подключиться к сессии демона
./agents.sh stop       # мягко: STOP-флаг, новые задачи не раздаются
./agents.sh stop --now # жёстко: остановить демон и агентов
```

Витрина — `http://localhost:7777`: таблица агентов, журнал, кнопки «Тик сейчас» и STOP.

## Как устроен тик

Тик — **отдельный процесс** `claude -p /orchestrate`, который умирает вместе с ответом.
Не `/loop`: тот живёт внутри одной сессии, накапливает контекст и рано или поздно
компактится — получается частичная память, а это хуже полного её отсутствия.
Здесь накапливать нечего, и стоимость тика не растёт со временем.

Оркестратор ходит в `gh` сам: читает issues, лейблы, PR. **MCP про задачи не знает** —
его словарь только про агентов:

| Инструмент | |
| --- | --- |
| `list_agents` | кто жив: status, `idleMin`, alive, worktree |
| `spawn_agent` | запустить роль с готовой задачей (или отказать по потолкам) |
| `stop_agent` | мягко остановить и освободить слот |
| `agent_log` | выжимка последних действий агента |

## Три ручки расширения

1. **Механика** — `.claude/skills/orchestrate/SKILL.md`, проза. Перечитывается каждый тик.
2. **Новая роль** — новый `.claude/skills/<имя>/SKILL.md` + строчка в `roles` конфига.
3. **Настройки** — `limits`, `dryRun`, `tickIntervalMin` в `config.jsonc`. Перечитываются
   каждый тик; рестарт нужен только при смене портов.

## Контракт входа скилла

Задача попадает в агента **только** JSON-файлом, путь передаётся аргументом:

```json
{ "id": "issue-136", "title": "…", "fields": { "issue": 136 }, "text": "вся постановка" }
```

Скилл не ходит за постановкой никуда: что положили в `text`, с тем он и работает.
Отсюда же отладка — файл пишется руками:

```sh
./agents.sh run implement task.json              # здесь же, видит незакоммиченный скилл
./agents.sh run implement task.json --worktree   # как в проде, в отдельном дереве
./agents.sh run orchestrate                      # один тик (нужен живой демон — он держит MCP)
```

## Что важно знать

- **Скиллы должны быть в `main`.** `claude --worktree` создаёт дерево от `main`, а не от
  текущей ветки — незамёрженный `.claude/skills/*` до спавненного агента не доедет.
  Поэтому отладочный `run` по умолчанию работает без отдельного worktree.
- **Агент не завершается сам.** Доделав работу, он переходит в `idle` и держит слот, пока
  его не остановят. Это забота оркестратора; `stop_agent` мягкий, диалог сохраняется
  (`claude attach <id>` поднимает обратно).
- **Ошейник — это `--tools`, а не `--allowedTools`.** Второй раздаёт разрешения, но набор
  инструментов не сужает: без `--tools` оркестратору видны Write, WebFetch и прочее.
- **Демон не владеет правдой.** Реестра агентов в памяти нет: состояние выводится из
  `claude agents --json` и mtime сессий, счётчик спавнов — из `history.jsonl`. Рестарт демона
  ничего не восстанавливает, а агенты (запущены через `--bg`) его смерть переживают.
- **Юрисдикция машинерии** — только фоновые сессии внутри `.claude/worktrees/`.
  Интерактивные сессии не трогаются никогда, даже при `stop --now`.
- **Потолки живут в `spawn_agent`**, а не в прозе скилла: инструкцию модель себе уговорит,
  отказ инструмента — нет. `dryRun: true` — режим наблюдения: думает и пишет решения, но
  никого не запускает.
