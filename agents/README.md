# @vexx/agents

Запуск агентов Vexx: роли, планировщик, MCP и витрина.

Доска GitHub и её состояния живут в соседнем пакете [`project-config/`](../project-config) —
здесь про неё нет ничего.

## Главная идея: все агенты равноправны

Оркестратор, реализатор, будущий аналитик — это **одинаковые агенты**, запущенные одним
и тем же кодом (`src/launch.ts`). Всё, чем они отличаются, описано полями роли в
`config.jsonc`. Всем без исключения подкладывается один и тот же MCP-сервер.

Роль = строчка в `config.jsonc` + каталог в `.claude/skills/`. Больше ничего.

## Ключ агента

Из пары (роль, аргумент) выводится всё:

```
implement + 181  →  ключ implement-181
                 →  окно tmux   agents:implement-181
                 →  worktree    .claude/worktrees/implement-181
                 →  сессия      uuidv5("implement-181") — навсегда одна и та же
```

Реестра мы не заводим: **агент — это окно tmux**, его имя и есть ключ. Живость — окно
существует, остановка — `kill-window`, перехват — `tmux attach`. Связка «задача ↔ агент»
видна, а не хранится.

## Create-or-update: как позвать агента обратно

Повторный запуск с тем же аргументом **не создаёт нового агента**, а продолжает разговор
с прежним — он помнит всё, что делал. Так его и зовут после ревью:

```sh
./agents.sh spawn probe alpha   # первый раз  → --session-id <uuid>, новое дерево
./agents.sh wake  probe alpha   # потом       → --resume <тот же uuid>
```

`wake` и `spawn` — одна и та же команда: выбора между «создать» и «продолжить» у
вызывающего нет, это решает `launch.ts` по наличию дерева. К **живому** агенту
дозапуститься нельзя — будет отказ со ссылкой на `watch`: его сессия сейчас открыта
в его окне, и второй claude на неё пускать нельзя.

Следствие: **дерево агента живёт, пока живёт задача**. Убирается руками:

```sh
git worktree remove -f -f .claude/worktrees/implement-181   # claude запирает своё дерево
git branch -D worktree-implement-181
```

## Почему интерактивный claude в tmux, а не `--bg` и не `-p`

Задать свой `--session-id` — единственный способ звать агента обратно, не ведя реестра.
Проверено, чем это оплачивается:

| | `--bg` | `-p` | **tmux** |
|---|---|---|---|
| наш `--session-id` | игнорируется | ✓ | **✓** |
| `--resume` без форка | форкается | ✓ | **✓** |
| `status` из `claude agents --json` | ✓ | ✗ | **✓** |
| переживает рестарт сервера | ✓ | ✗ | **✓** |
| перехватить руками | `claude attach` | ✗ | **`tmux attach`** |
| завершается сам, доделав | ✗ | ✓ | ✗ |

`--bg` чеканит свой id и при `--resume` форкается — цепочку id не предсказать.
`-p` живёт один вызов. Интерактивный в tmux даёт всё сразу, ценой того, что доделавший
агент не завершается сам: место освобождает оркестратор, `stop_agent` → `kill-window`.

Диалог доверия (`Is this a project you trust?`), который ловит интерактивный claude
в незнакомом каталоге, здесь не возникает: запуск идёт из доверенного корня репозитория,
а дерево claude заводит себе сам флагом `--worktree`.

## Команды

```sh
./agents.sh start [--paused]      сервер в tmux: планировщик + MCP + витрина
./agents.sh status | logs | attach
./agents.sh watch <ключ>          подключиться к живому агенту и договорить руками
./agents.sh stop [--now]          мягко (STOP) или жёстко (плюс все агенты)

./agents.sh run   <роль> [арг]    форграунд, живой диалог — отладка скилла руками
./agents.sh spawn <роль> [арг]    фоном, как по расписанию
./agents.sh wake  <роль> <арг>    синоним spawn
./agents.sh list | stop-agent <ключ>
./agents.sh tick                  запустить orchestrate прямо сейчас
```

Планировщик, MCP и CLI ходят через один и тот же `launch()`, поэтому отлаживается ровно то,
что потом поедет по расписанию.

## Поля роли

| Поле | Смысл |
| --- | --- |
| `skill` | каталог в `.claude/skills/`, он же команда `/<skill>` |
| `mode` | `oneshot` — `-p`, отработал и умер; `session` — живёт в окне tmux с памятью |
| `worktree` | своё git-дерево (иначе агенты подерутся за ветку рабочей копии) |
| `everyMin` | запускать по расписанию; нет поля — только по требованию |
| `tools` | значение `--tools`; `"default"` — не сужать набор инструментов |
| `allow` | значение `--allowedTools` |
| `permissionMode` | `acceptEdits` для форграунда, `bypassPermissions` для фона |

`--tools` и `--allowedTools` — разные вещи: первый сужает **набор** встроенных инструментов,
второй лишь раздаёт разрешения. Без `--tools` оркестратору были видны `Write`, `WebFetch`
и `Task*` — проверено на прошлой версии машинерии.

## Инструменты MCP

Выдаются **всем** ролям: `list_agents`, `spawn_agent({role, arg})`, `stop_agent({key})`,
`agent_log({key, limit})`. Про issue и доску MCP не знает ничего — за задачами агенты
ходят в `gh` сами.

## Мёрж по `/approved`

Человек пишет в комментарии PR `/approved` — и дальше PR доводит до мёржа **тот же агент,
который его делал**: ребейз на свежий `main`, зелёный CI, `gh pr merge --squash`, задача
в `Done`. Оркестратор только замечает одобрение и будит агента (`spawn_agent` с тем же
номером задачи — то есть продолжение прежнего разговора).

Почему комментарий, а не штатное «Approve»: агенты ходят под тем же аккаунтом, PR создан
от вашего имени, а GitHub **не даёт апрувить собственные PR**. Кнопки ревью просто нет.

Из этого же следует граница, которую нечем подпереть технически: агент может написать
`/approved` сам — токен один и тот же. Запрет живёт в `SKILL.md`, как и потолки. Настоящее
решение — отдельный аккаунт для агентов, и оно отложено сознательно.

Чтобы одно и то же одобрение не будило агента каждый тик, оркестратор помнит отметку
`approvedSince` в `.agents-runs/state/orchestrate.json` и двигает её **только за фактически
разбуженным агентом** — иначе одобрение потерялось бы навсегда.

## Аналитик и колонка Analytics

Перед `To implement` на доске стоит `Analytics`: человек кладёт туда задачу в пару строк,
аналитик (`analyst-<issue>`) разворачивает её в постановку и обсуждает в комментариях issue.
Из `Analytics` в `To implement` задачу двигает **только человек**, приняв разбор.

Аналитик долгоживущий (как реализатор — `mode: session`), но с двумя особенностями:

- **Живёт ~30 минут в цикле**, отвечая на комментарии сразу, а не раз в тик. Ждёт
  блокирующим `sleep`, поэтому во время ожидания он `busy` — оркестратор его не трогает.
- **Уснувшего поднимает оркестратор** по новому комментарию, по образцу `/approved`:
  отметка `analyticsSeen` в `state/orchestrate.json`, движется только за поднятым аналитиком.

Потолки реализаторов и аналитиков **раздельные** (по 2 каждого) — аналитик не вытесняет
реализацию. Всё это — прозой в скиллах, роль `analyst` в конфиге кода не потребовала.

## Обновление скиллов доходит не сразу

`orchestrate` работает в рабочей копии (`worktree: false`) — правки его скилла действуют
немедленно. А session-агенты (`implement`, `analyst`) читают скилл из своего дерева,
замороженного на момент его создания: обновление доедет до них только с **новым** деревом.
Поменяли скилл живущему агенту и нужно, чтобы он его увидел, — пересоздайте дерево:

```sh
./agents.sh stop-agent <ключ>
git worktree remove -f -f .claude/worktrees/<ключ> && git branch -D <ветка>
./agents.sh spawn <роль> <номер>     # создаст свежее дерево от origin/main
```

## Лимитов в коде нет

Осознанное решение: ограничения (не больше двух агентов, один агент на задачу) описаны
**прозой** в `.claude/skills/orchestrate/SKILL.md`. `spawn_agent` не отказывает никогда.
Цена ошибки в тексте — счёт за токены, так что за журналом стоит поглядывать.

Потолки считаются **только по `list_agents`**, то есть по происходящему сейчас. Считать их
по журналу нельзя: туда попадают запуски руками, пробные роли и тики самого оркестратора —
и он принимает чужую отладку за свою активность, отказываясь от работы на ровном месте.

## Журнал

`.agents-runs/history.jsonl`, append-only. Пишется **только** в `launch.ts`, через который
проходит любой запуск, — мимо журнала запустить агента нельзя. В записи есть `trigger`
(`schedule` / `mcp` / `cli` / `dashboard`), `by` (кто дёрнул) и полная командная строка,
которую можно скопировать и повторить руками. Витрина показывает последние 20 записей.

## Что важно знать

- **Скилл должен быть ЗАПУШЕН.** `claude --worktree` отводит дерево от `origin/main`, а не
  от локального — закоммиченный, но не отправленный скилл до агента не доедет, и тот
  встретит `Unknown command`. Поэтому же сервер раз в 10 минут делает `git pull --rebase`
  на `main`.
- **Есть роль `probe`** — проверить механику запуска, ничего не делая: `./agents.sh spawn
  probe alpha`, потом ещё раз то же самое. Первый ответ — «ПЕРВЫЙ ЗАПУСК», второй —
  «ПРОДОЛЖЕНИЕ, первая метка alpha». Если второй говорит «первый запуск» — сломался resume.
- **Агент сам не завершается**: доделав работу, он уходит в `idle` и держит окно, пока его
  не остановят. Это работа оркестратора.
- **Посмотреть на живого агента**: `./agents.sh watch implement-181` — настоящий терминал,
  можно вмешаться и договорить руками.
- **Сервер не владеет правдой.** Состояние выводится из мира заново, агенты запущены
  отсоединённо, поэтому рестарт сервера ничего не рвёт и ничего не восстанавливает.
- **Конфиг — на горячую.** `config.jsonc` перечитывается с диска на каждом обращении
  (витрина, MCP, планировщик). Добавил роль, поменял `everyMin`, поправил `tools` — эффект
  сразу, без рестарта. Рестарт нужен **только** под смену портов: сокеты биндятся на старте.
- **Каждый призыв порождает новый файл сессии** с копией прошлой переписки внутри.
